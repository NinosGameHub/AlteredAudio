# VST Effect Engine Roadmap
### One plugin — modular rack — full signal routing — Windows / VST3

---

## Overview

You are building a single VST3 plugin that acts as a **modular effects rack**. Users load it into Cubase or Bitwig, then build signal chains by adding, reordering, and connecting DSP modules (filters, reverb, distortion, compression, modulation) inside a custom UI. Modules can be chained serially or blended in parallel. The whole chain is saveable as a preset.

**Recommended framework: JUCE 8**
JUCE handles the VST3 SDK wrapping, audio threading, parameter management, UI rendering, and cross-buffer-size audio processing — all in one library. It is the industry standard for exactly this kind of project. With zero C++ experience, using raw VST3 SDK would add months of unnecessary pain. JUCE is the right call.

*Goal: get a blank JUCE plugin compiling and loading in Cubase/Bitwig*

- [ ] **Visual Studio 2022 Community** — free C++ compiler for Windows

- [ ] **Git** — version control (`git-scm.com`)
- [ ] **JUCE 8** — clone from `github.com/juce-framework/JUCE`
- [ ] **VST3 SDK** — included inside JUCE, no separate install needed

### 0.2 Project Scaffold
- [ ] Create a JUCE `AudioProcessor` project via CMake (not Projucer — CMake is cleaner with Claude Code)
- [ ] Confirm it compiles to a `.vst3` file
- [ ] Copy `.vst3` to `C:\Program Files\Common Files\VST3\`
- [ ] Scan and load it in Cubase and Bitwig — confirm it appears with no crash

### 0.3 Learn Exactly Enough C++
You don't need to master C++. You need to recognise these patterns to work with Claude Code effectively:
- Pointers and references (`*`, `&`)
- Classes and inheritance
- `std::vector`, `std::unique_ptr`, `std::shared_ptr`
- `float` vs `double` (audio always uses `float` for performance)
- Header files (`.h`) vs implementation files (`.cpp`)

**Resource:** *The Audio Programmer* on YouTube — JUCE-specific tutorials, free.

## Phase 1 — Core Audio Engine
*Goal: audio flows through the plugin with no modules yet — just passthrough*

### 1.1 Audio Thread Architecture
- [ ] Implement `prepareToPlay(sampleRate, blockSize)` — called by DAW before playback
- [ ] Implement `processBlock(AudioBuffer, MidiBuffer)` — called per buffer, this is the hot path
- [ ] Handle all buffer sizes (64 to 2048+ samples) without assumptions
- [ ] Support 44.1kHz and 48kHz — store sample rate, pass it to every module

### 1.2 Module Base Class
Design a `EffectModule` base class that every effect inherits from:

```
EffectModule
  ├── prepare(sampleRate, blockSize)   — called once before playback
  ├── process(AudioBuffer)             — called every buffer
  ├── reset()                          — clear internal state (tails, delays)
  ├── getParameters()                  — returns list of automatable params
  └── getName()                        — "Filter", "Reverb", etc.
```

- [ ] Build the abstract base class
- [ ] Write a `PassthroughModule` as the first concrete implementation
- [ ] Confirm audio passes through it cleanly

### 1.3 Module Chain
- [ ] `ModuleChain` class — owns an ordered list of `EffectModule` instances
- [ ] `processChain(AudioBuffer)` — iterates modules in order, pipes audio through each
- [ ] Support bypass per module (skip processing, pass audio through)
- [ ] Support serial chain (default) and parallel blend (wet/dry mix between two branches)

### 1.4 Channel Handling
- [ ] Stereo (L/R) — standard, process both channels
- [ ] Mono — process single channel, duplicate to both outputs
- [ ] Mid/Side — encode to M/S before chain, decode after
  - Mid = (L + R) / 2
  - Side = (L - R) / 2

## Phase 2 — DSP Modules
*Goal: build each effect module as a self-contained class*

Every module follows the same pattern: `prepare → process → reset`. All DSP math can be generated by Claude Code — you direct the structure, Claude writes the formulas.

### 2.1 Filters
- [ ] **Biquad filter core** — the foundation of all filter types
  - Lowpass, highpass, bandpass, notch, allpass, peak, shelf
  - Parameters: frequency (20Hz–20kHz), Q/resonance, gain (for peak/shelf)
- [ ] **EQ module** — multiple biquad bands in series (parametric EQ)
  - Up to 8 bands, each independently configurable
- [ ] Sample-rate-aware coefficient recalculation on parameter change

### 2.2 Reverb / Delay
- [ ] **Delay module**
  - Parameters: time (ms), feedback, wet/dry, ping-pong toggle
  - Tempo sync to DAW BPM via JUCE's `AudioPlayHead`
- [ ] **Reverb module**
  - Algorithm: Schroeder/Moorer or convolution (start with algorithmic)
  - Parameters: room size, damping, pre-delay, wet/dry
  - Tail handling — reverb must continue after plugin bypass

### 2.3 Distortion / Saturation
- [ ] **Waveshaper module**
  - Algorithms: soft clip, hard clip, tanh saturation, bit crush, fold-back
  - Parameter: drive, tone (post filter), wet/dry
- [ ] **Oversampling** — run distortion at 2x or 4x sample rate to reduce aliasing
  - JUCE provides `dsp::Oversampling` — use it

### 2.4 Compression / Dynamics
- [ ] **Compressor module**
  - Parameters: threshold, ratio, attack, release, knee, makeup gain
  - Gain reduction meter output (for visualizer)
- [ ] **Limiter module** — compressor with infinite ratio, fast attack
- [ ] **Gate module** — downward expander, silences signal below threshold

### 2.5 Modulation
- [ ] **LFO engine** — internal, runs per-buffer
  - Shapes: sine, triangle, square, sawtooth, random (S&H)
  - Parameters: rate (Hz or tempo-sync), depth, phase
- [ ] **Chorus module** — short modulated delay (5–30ms), multiple voices
- [ ] **Flanger module** — very short modulated delay (0.5–5ms) with feedback
- [ ] **Phaser module** — all-pass filter cascade modulated by LFO

## Phase 3 — Parameter & Modulation System
*Goal: every knob is automatable, modulatable, and thread-safe*

### 3.1 Parameter Registration
- [ ] Use JUCE's `AudioProcessorValueTreeState` (APVTS) — the standard way
- [ ] Every module registers its parameters with unique IDs (`filter1_freq`, `reverb1_size`)
- [ ] Parameters are automatable in Cubase and Bitwig automatically via APVTS

### 3.2 DAW Automation
- [ ] All parameters exposed to DAW via APVTS — no extra work needed once registered
- [ ] Test automation lanes in Cubase and Bitwig

### 3.3 Internal LFO Modulation
- [ ] `ModulationMatrix` class — maps LFO outputs to parameter targets
- [ ] Each parameter can have a modulation depth (-1.0 to +1.0)
- [ ] LFO value added to base parameter value every buffer

### 3.4 Envelope Followers
- [ ] `EnvelopeFollower` class — tracks amplitude of audio signal
- [ ] Attack/release smoothing
- [ ] Output mapped to parameters via ModulationMatrix (same as LFO)

### 3.5 Thread Safety
- [ ] Audio thread never allocates memory — all allocations in `prepareToPlay`
- [ ] Parameter changes from UI thread use `std::atomic` or APVTS's built-in thread safety
- [ ] Never call UI code from the audio thread

## Phase 4 — Preset System
*Goal: save and recall full plugin state and individual module states*

### 4.1 State Serialization
- [ ] Use JUCE's `ValueTree` — serializes to XML/binary
- [ ] Full plugin state = module chain order + all parameter values + routing config
- [ ] Per-module state = just that module's parameters

### 4.2 Preset Storage
- [ ] Factory presets — bundled with plugin, read-only
- [ ] User presets — saved to `%APPDATA%\YourPlugin\Presets\`
- [ ] Preset format: `.xml` (human-readable, easy to debug)

### 4.3 Preset Browser
- [ ] List view in UI — shows factory and user presets
- [ ] Save / Load / Delete / Rename
- [ ] Per-module preset save — save just one module's settings to reuse in other chains

## Phase 5 — Custom UI
*Goal: a rack-strip UI where modules are visible, reorderable, and visually themed*

### 5.1 JUCE UI Foundation
- [ ] `AudioProcessorEditor` — the plugin window
- [ ] Use JUCE's `Component` system for all UI elements
- [ ] Set a fixed or resizable window size (recommended: 900×600 minimum)

### 5.2 Rack Strip Layout
- [ ] **Module strip** — horizontal or vertical rack showing loaded modules in order
- [ ] Each module renders as a **card/panel** with its own knobs and visualizer
- [ ] **Drag to reorder** — JUCE's `DragAndDropContainer` handles this
- [ ] **Add module** button — opens a picker showing all available module types
- [ ] **Remove module** button per card
- [ ] **Bypass toggle** per card (greys out the card visually)
- [ ] **Serial/Parallel toggle** between adjacent modules

### 5.3 Per-Module Visualizers
- [ ] **EQ module** — frequency response curve drawn with `Path` (JUCE graphics)
- [ ] **Compressor** — gain reduction meter (vertical bar, updates every ~60ms)
- [ ] **Reverb/Delay** — simple wet level meter
- [ ] **Distortion** — input/output waveform or transfer curve
- [ ] **Modulation** — LFO shape display, current phase position

### 5.4 Custom Theme System
- [ ] `ThemeManager` class — stores colour palette, font choices, knob style
- [ ] Themes defined as data (JSON or ValueTree) — no code changes to switch
- [ ] Ship with 2–3 default themes (dark, light, warm)
- [ ] User-editable theme via a colour picker panel
- [ ] All components read colours from `ThemeManager` — no hardcoded hex values

### 5.5 Knobs & Controls
- [ ] Custom `RotarySlider` look — JUCE's `LookAndFeel` override
- [ ] Right-click on any knob → context menu: Reset to default / Set modulation / Enter value
- [ ] Modulation depth shown as coloured arc around knob

## Phase 6 — Testing & DAW Integration
*Goal: stable, no-crash, no-glitch operation in Cubase and Bitwig*

- [ ] Run **JUCE's AudioPluginHost** — a test host included with JUCE

- [ ] Load plugin, confirm scan passes
- [ ] Test automation recording and playback
- [ ] Test preset save/load via DAW's plugin state save
- [ ] Test at 44.1kHz and 48kHz
- [ ] Test at multiple buffer sizes (64, 128, 256, 512, 1024)

### 6.3 Bitwig Testing
- [ ] Same checklist as Cubase
- [ ] Bitwig has stricter VST3 conformance — run validator first

### 6.4 Stress Testing
- [ ] Load all modules simultaneously
- [ ] Automate every parameter at once

- [ ] Check CPU usage — target under 5% with all modules active at 44.1kHz / 256 samples

## Recommended Build Order

```
Phase 0 → Phase 1 → Phase 2 (one module at a time) → Phase 3 → Phase 4 → Phase 5 → Phase 6 → Phase 7
```

Don't start Phase 5 (UI) until Phase 2 is working. A beautiful UI on broken DSP is wasted time.

Start Phase 2 with the **Filter module** — it is the simplest DSP, uses the biquad pattern everything else builds on, and gives you a working visualizer early.

## Working with Claude Code

Claude Code is well-suited to this project for:
- Generating boilerplate JUCE class scaffolds
- Writing DSP math (biquad coefficients, compressor gain curves, LFO shapes)
- Debugging CMake configs
- Writing the serialization / preset code
- Generating `LookAndFeel` UI overrides

You direct the architecture. Claude writes the implementation. Review every audio-thread function Claude writes for allocations — the golden rule is **no `new`, no `malloc`, no STL resizing on the audio thread**.
